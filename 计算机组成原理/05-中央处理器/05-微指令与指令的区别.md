# 微指令与指令的区别

## 定义

- **指令**（机器指令）：用户编程的基本单位，如 `ADD R1, R2`。存放在主存中，由程序员/编译器生成。
- **微指令**：微程序控制器中一个时钟周期内发出的一组控制信号。存放在 CPU 内的控制存储器（CM）中，由硬件设计者定义，对程序员透明。

> 一句话：**一条机器指令由一段微程序（若干条微指令）解释执行**。

---

## 核心对比

| 维度 | 指令（机器指令） | 微指令 |
|---|---|---|
| 层次 | 软件可见的最底层 | 硬件实现层，对软件透明 |
| 存放位置 | 主存（RAM） | 控制存储器 CM（CPU 内部 ROM） |
| 取出依据 | PC → MAR → 主存 | μPC → CMAR → 控制存储器 |
| 功能 | 完成一次完整的运算/传送/转移 | 产生一拍控制信号（打开某组门电路） |
| 格式 | 操作码 + 地址码 | 控制字段（每位/每组对应一个控制信号） |
| 设计者 | 程序员 / 编译器 | CPU 硬件设计者 |
| 变化频率 | 随程序不同而变化 | 出厂后固定（微程序 ROM 一般不可改） |

---

## 核心定理 / 性质

### 微指令的两种编码方式

| | 水平型微指令 | 垂直型微指令 |
|---|---|---|
| 编码方式 | 控制字段每位对应一个控制信号，直接驱动硬件 | 控制字段经译码后产生控制信号 |
| 并行度 | 高（一拍可同时发出多个微操作） | 低（一拍通常只做一件事） |
| 微指令长度 | 长（控制信号数量 = 位数） | 短（用编码压缩） |
| 微程序长度 | 短（并行做更多事） | 长（需要更多微指令） |
| 设计难度 | 复杂（需精心安排并行操作不冲突） | 简单（类似机器指令设计） |
| 典型应用 | 追求速度的 CISC CPU | 追求紧凑的微架构 |

> 实际 CPU 常采用**混合型**：对互斥的微操作用编码压缩，对可并行的微操作用直接控制位。

### 指令与微指令的执行关系

```
                    ┌── 一个指令周期 ──┐
机器指令层:   取指  ─┬─  间址  ─┬─  执行  ─┬─  中断
                    │          │          │
微指令层:     μI1, μI2, μI3   μI4, μI5   μI6, μI7, μI8 ...
              └── 取指微程序 ──┘           └── 执行微程序 ──┘
```

每条机器指令对应一段**微程序**。以 `ADD R1, (addr)` 为例，上层是一条机器指令，下层可能对应 8~10 条微指令，每条微指令在一个时钟周期发出若干控制信号：PC→MAR、读内存、MDR→IR、R1→Y、读内存→MDR→ALU、Z→R1 等。

## 典型例题

**例**：判断下列说法正误。

(1) "微指令就是比机器指令更短的指令" —— ❌ 错误。微指令不是"短指令"，而是不同层次的概念。微指令控制的是硬件门电路，机器指令完成的是算法语义。

(2) "一台计算机的指令系统变了，微程序也必须重新设计" —— ✔ 正确。每条指令由一段微程序解释，指令系统改变意味着全部微程序要重新编写并烧录到控制存储器中。

(3) "微程序控制器比硬布线控制器慢" —— ✔ 正确。微程序控制器执行一条机器指令需要多次访问控制存储器（取微指令），而硬布线控制器直接通过组合逻辑产生控制信号，速度快但不可修改。

## 易错点

1. **微指令、微命令、微操作的区分**：
   - **微操作**：最底层的硬件动作（如 "PC→MAR"）
   - **微命令**：控制微操作的控制信号（如 "PCout, MARin"）
   - **微指令**：同一时钟周期内一组微命令的集合
   - 三者关系：一条**微指令**包含多个**微命令**，每个微命令控制一个**微操作**

2. **控制存储器（CM）不是主存**：CM 在 CPU 内部，由 ROM 实现（通常是 EPROM/EEPROM），微程序烧录其中。μPC 访问 CM 的过程独立于 PC 访问主存的过程——两者有各自的地址寄存器和数据通路。

3. **并非所有 CPU 都使用微指令**：硬布线控制器不使用微程序，直接通过组合逻辑电路根据指令的操作码和节拍产生控制信号。RISC 多为硬布线，CISC 多采用微程序。

4. **微程序 ≠ 汇编程序**：微程序是 CPU 内部解释指令的硬件级代码，汇编程序是软件层面的低级语言。两者毫无关系。
