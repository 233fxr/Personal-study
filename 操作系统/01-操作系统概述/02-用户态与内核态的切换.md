# 用户态与内核态的切换

## 定义

现代 OS 依赖 CPU 的双模式（dual mode）机制来保护系统：

- **用户态（目态）**：只能执行非特权指令，访问受限的地址空间。用户程序在此运行。
- **内核态（管态）**：可执行所有指令，访问全部地址空间。OS 内核在此运行。

CPU 通过程序状态字（PSW）中的一位模式标志来区分当前状态。

> 核心原则：用户程序想做任何涉及硬件资源的事（读文件、申请内存、网络通信），必须主动或被动地"坠入"内核态，由 OS 代为执行，执行完再"弹回"用户态。

---

## 用户态 → 内核态（三种触发方式）

### 1. 系统调用（主动）

进程主动请求 OS 服务时，执行访管指令（`trap` / `int` / `syscall`），触发软中断，CPU 切换到内核态，跳转到系统调用处理程序。

例：`read()` 读文件——用户程序不能直接操作磁盘，必须通过系统调用让内核去读。

### 2. 中断（被动，来自外部）

外部硬件设备（时钟、键盘、网卡、磁盘）通过中断控制器向 CPU 发出中断请求信号 INTR。CPU 在当前指令执行完毕后检查 INTR，若 IF=1（开中断），则响应中断：保存现场 → 切内核态 → 执行中断服务程序。

例：时钟中断——每次时钟滴答，CPU 进入内核态执行调度检查，决定是否切换进程。

### 3. 异常（被动，来自内部）

CPU 执行指令过程中检测到错误或特殊条件，自身触发异常。异常的处理在内核态进行。

例：缺页异常（访问的页面不在内存）→ 进入内核态的缺页处理程序 → 从磁盘调入页面 → 返回用户态重试指令；除零异常 → 内核向进程发 SIGFPE 信号。

| 触发方式 | 来源 | 主动/被动 | 典型例子 |
|---|---|---|---|
| 系统调用 | 程序 | 主动 | `read`, `fork`, `malloc` 申请堆内存 |
| 中断 | 外设 | 被动 | 时钟滴答、键盘敲击、网卡收包 |
| 异常 | CPU 自身检测 | 被动 | 缺页、除零、非法指令、断点 |

---

## 内核态 → 用户态（返回）

内核态切回用户态总是发生在中断/异常/系统调用的**返回阶段**：

1. 内核完成服务后，执行**中断返回指令**（如 x86 的 `IRET`）
2. `IRET` 从内核栈弹出之前保存的 PSW 和 PC
3. PSW 的模式位被恢复为"用户态"，PC 恢复到用户程序的断点
4. CPU 继续执行用户程序的下一条指令

另外，进程调度也会涉及切换：当内核决定让一个新的用户进程开始运行时，该进程首次获得 CPU 时的状态就是**用户态**——新进程的内核栈上被提前构造好了一个"假装刚从中断返回"的栈帧，`IRET` 后直接进入用户态执行。

## 典型例题

**例**：下列操作中，哪些必然导致 CPU 从用户态切换到内核态？

① 执行 `printf` 输出字符串　② 访问一个不在内存中的页面　③ 执行整数加法　④ 网卡收到一个数据包

答：①②④。

① 涉及系统调用 `write`，切内核态。
② 触发缺页异常，切内核态处理。
③ 普通运算，不涉特权指令，保持在用户态。
④ 网卡收包触发硬件中断，CPU 响应中断时切内核态。注意：中断到达时 CPU 不一定马上响应（若关中断），但一旦响应就必切内核态。

## 易错点

1. **系统调用不是普通的函数调用**：`read()` 本质上是一条 `syscall` 指令 + 内核态的 OS 代码，不是跳转到另一个用户态函数。调用过程伴随完整的上下文保存和栈切换。

2. **缺页异常发生在指令执行中途**：比如 `MOV EAX, [0x1234]` 这条指令执行到 MEM 阶段发现缺页，在**该指令未完成**时触发异常。内核处理完缺页后，返回用户态时**重新执行这条指令**（不是执行下一条）。这和中断不同——中断总是在指令边界响应。

3. **用户态 ↔ 内核态的切换是有开销的**：每次切换都要保存寄存器、切换栈、刷新 TLB 部分项。这就是为什么频繁的系统调用会拖累性能——`epoll` 之类的机制正是为了解决"每个连接一个 `read` 调用"的高开销问题。

4. **不是所有中断都导致进程切换**：中断一定会切内核态，但不一定换进程。时钟中断才可能触发调度，I/O 完成中断通常只是唤醒等待进程而不立即切换。
