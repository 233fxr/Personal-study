# 碎片整理与动态重定位

## 定义

**外部碎片**：各已分配分区之间存在足够总容量的空闲空间，但因不连续，无法满足新进程的需求。

**紧凑（Compaction，碎片整理）**：通过移动已在内存中的进程，将分散的空闲空间合并为大块连续区域。要求系统具备**动态重定位**能力——进程被移动后，其逻辑地址到物理地址的映射能被即时更新，进程感知不到自己被移动了。

**动态重定位**：程序执行期间，CPU 每次访存时都将逻辑地址实时转换为物理地址，转换规则可以动态修改，使程序能被整体平移。

---

## 核心机制

### 重定位寄存器方案（连续分配时代）

```
     逻辑地址 100
         │
         ▼
    ┌─────────┐     ┌──────────┐
    │  CPU    │────►│ 重定位寄存器 │  (存放当前进程的物理起始地址)
    │         │     │  base = 5000 │
    └─────────┘     └────┬─────┘
                         │
                         ▼
            物理地址 = 逻辑地址 + base = 5100
                         │
                         ▼
                    ┌─────────┐
                    │   主存    │
                    └─────────┘
```

物理地址 = 逻辑地址 + base（重定位寄存器值）。

**紧凑的过程**：OS 暂停进程 → 将其占据的物理内存块整体拷贝到新位置 → 修改该进程的重定位寄存器 base 值 → 恢复执行。进程内部的逻辑地址完全不变，且不能感知自己被移动了——`MOV AX, [100]` 取到的仍然是正确的数据。

### 页表方案（分页时代）

分页系统中紧凑不需要移动整个进程——只需**移动若干页面**，然后修改页表中对应页表项的页框号。粒度更小，效率更高。

### 分段方案

移动段后，修改段表中该段的基址（base）字段。逻辑上相当于每段有一个重定位寄存器。

## 紧凑的条件

| 条件 | 说明 |
|---|---|
| 动态重定位硬件支持 | 地址转换在运行时可改（重定位寄存器 / 页表 / 段表） |
| 进程可暂停 | OS 能挂起进程，等待拷贝完成 |
| 地址绑定时机为执行期 | 编译时或装入时绑定的地址无法重定位——必须执行期动态绑定 |
| 拷贝开销可接受 | 大量数据的 memcpy 消耗 CPU 和总线带宽 |

> 现代 OS 基本不在物理内存层面做紧凑——分页机制本身就消灭了外部碎片（任何空闲页框都能分配给任何页面），紧凑的需求转移到了**磁盘碎片整理**（文件系统层面）。

## 典型例题

**例**：某连续分区系统中，重定位寄存器的 base 值为 0x8000。现 OS 决定将该进程平移到物理地址 0xC000 处。说明操作过程。

解：
1. 暂停该进程
2. 将原物理地址 0x8000～0x8000+进程大小 的内存块整体拷贝到 0xC000
3. 将重定位寄存器 base 值由 0x8000 改为 0xC000
4. 恢复执行。进程中所有逻辑地址（如 0x0100）经新的 base 重定位后物理地址变为 0xC100，数据全部正确

## 易错点

1. **重定位寄存器的值不是"偏移量"的概念**：逻辑地址 0 对应物理地址 base。所以 base 就是进程的物理起始地址。

2. **紧凑 ≠ 分页**：分页通过离散分配消灭外部碎片，根本不需要紧凑。紧凑针对的是**连续分配**方案，考试中出现"紧凑"基本暗示是连续分区分配。

3. **动态重定位 ≠ 动态链接**：动态重定位是地址映射层面的（逻辑→物理），动态链接是程序加载层面的（符号→地址）。两者都带"动态"但完全两回事。

4. **紧凑时进程不能继续执行**：进程的内存内容正在被拷贝，此时若继续执行可能读到半拷贝状态的数据。必须挂起等待。
